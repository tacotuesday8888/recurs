# Sub-Agent Harness Comparison

Verified against current official open-source repositories on 2026-07-18 and pinned to exact commits below. This comparison describes execution mechanics, not marketing claims or source copying. Roo Code is included as a historical baseline because its pinned [repository notice](https://github.com/RooCodeInc/Roo-Code/blob/b867ec9145750d0ae1ff7f02d35406e9bf2a0b16/README.md) states that the Roo Code Extension was shut down on May 15.

## What the mature harnesses actually do

| Harness | Delegation and lifecycle | Authority, limits, and recovery | Recurs takeaway |
| --- | --- | --- | --- |
| Codex CLI | V2 exposes spawn, message/follow-up, wait, list, and interrupt over reusable child threads. Children can receive fresh, full, or last-N history, and terminal children can be unloaded from memory while their rollout remains available. | The effective parent approval policy, reviewer, permission profile, cwd, provider, and model configuration are copied; requested model/reasoning/service-tier overrides are catalog-validated. V2 defaults to four active threads including the root and permits a recursive task tree while bounding active execution and residency; unload/reload is supported within the live root session from materialized rollouts. Waiting is mailbox/event-driven. The inspected source does not establish full V2 tree restoration after process exit. | Separate durable task identity from active execution slots; make context policy and interrupt/follow-up semantics explicit; retain Recurs's stricter depth, spend, and workspace-import bounds. |
| OpenCode | `task` creates or resumes a parent-linked child session and returns a structured result. Experimental background jobs notify the parent and support cancellation. | Depth is configurable and defaults to one; agents can have step limits. A child can select its own model or inherit the parent. Parent deny/external-directory rules are carried down, but the child's own policy otherwise governs access. The background registry inspected is process-local and has no restart reconciliation. | Keep real child sessions and resumable IDs, but make effective authority a complete parent ceiling and persist every background transition. |
| Kimi Code | A session-scoped sub-agent host and swarm/batch tools create profiled children, track active work, emit lifecycle activity, and propagate cancellation. | Profiles choose prompt, tool set, model alias, and summary behavior; Explore and coding roles have different capabilities. Background task state is richer than a one-shot prompt, but Recurs should not infer restart safety without an explicit durable recovery contract. | Profiled swarms and visible activity are useful, but durable ownership, evidence, and recovery must be explicit rather than implied by the UI. |
| Kilo Code | Its OpenCode-derived `task` path creates parent-owned child sessions; Kilo adds a hardened task wrapper and explicit sub-agent presentation. | Depth is fixed to one, the task tool is removed from children, and child IDs are scoped to the current parent. The sandbox is intersected with the parent, while hard parent denies carry down for editing, shell/notebook execution, and configured MCP tools. Per-child models are selectable and cost deltas propagate upward. Its inspected experimental background mechanism remains in memory. | Preserve a complete monotonic `parent ceiling ∩ child policy` rule and upward cost accounting; do not equate an in-memory job with durable background work. |
| Roo Code | `new_task` pushes a sequential child onto an in-memory LIFO task stack while persisting parent/child histories and delegation metadata; nested chains are possible and completion returns control to the parent. | The child chooses a mode and inherits the active provider/profile. Tool access follows that mode; no explicit parent-policy intersection or maximum depth was found. Persisted histories and delegation metadata can be repaired/reopened, but there is no parallel/background supervisor. | Persist parent edges and repairable metadata, while keeping concurrency, depth, authority, and ownership explicit. Treat this as historical architecture, not an active product baseline. |
| Goose | The `summon` platform extension runs synchronous or asynchronous child agents with load/wait/cancel visibility. | Nested summon is forbidden; child turns default to 25 and asynchronous concurrency defaults to five. Child cwd must remain under the parent, models may inherit or override, and extensions are limited to the parent's set. Approvals are not forwarded, so children are forced to Auto. Handles/results are process-memory state and shutdown cancels them. | Adopt the explicit limits and extension subset, but never widen background approval to Auto and never claim restart durability for volatile handles. |
| Pi | The official repository has no first-class built-in sub-agent primitive; it ships an installable example extension that launches separate `pi` subprocesses in single, parallel, or chain modes. | The example caps eight tasks and four concurrent processes, accepts optional model/tool restrictions, runs `--no-session`, terminates on abort, and has no recovery. Implement/review is a prompt chain supplied by the example. | Keep orchestration in one internal engine rather than shelling Recurs out to itself. Prompt chains are inspiration, not a dependable review state machine. |
| Grok Build | `SubagentCoordinator` owns pending, active, and completed independent child sessions with fresh/forked/resumed context, foreground/background execution, result files, progress, cancellation, and optional worktrees. | Generic task depth is one. Explicit spawn, role, persona, then parent determine model/reasoning. Orphan reconciliation marks stale `running` work cancelled; completed results/transcripts can be loaded from disk. No generic concurrency, retry, or spend ceiling was found in the inspected path. Permission sharing is not a formal child≤parent lattice, and worktree failure can fall back to the shared workspace. | Keep Grok's provenance, context strategies, durable results, and honest orphan reconciliation; retain Recurs's explicit budgets and fail closed when required isolation cannot be created. |

## Codex Ultra correction

Codex really does define `ReasoningEffort::Ultra`. In the pinned source, Ultra selects proactive V2 multi-agent behavior when no custom hint overrides it. The inference client currently maps that local effort to wire-level `Max`. Therefore Ultra is a real local reasoning/orchestration setting, not a separate provider-side `ultra` wire value or evidence that child work survives a dead process.

## Current Recurs vertical

The source rows above are research findings. The implementation below is independently built against Recurs interfaces; no third-party source was pasted into it.

- `delegate_task` creates one exact-profile child; `delegate_tasks` runs a bounded ordered Explore/Review batch; `delegate_team` runs explicit Implement tasks, strict Review, and bounded finding-driven Repair.
- Every child is a separate pinned version-2 Recurs session running through the existing `ChildAgentManager`, `BackendRunCoordinator`, provider/runtime, tool, approval, continuation, and event seams.
- Child permissions cannot exceed the parent; children receive no delegation tools; depth is one and retries are zero. Version-4 team profiles expose no arbitrary-command or verification tools; only hardened Git inspection may spawn a process, and project scripts cannot be executed.
- Versioned Economy, Standard, Balanced, Performance, and Max policies freeze concurrency, children, requests, reported-cost ceiling, team width, reviewers, quality, and repair rounds. Version-1 through version-3 IDs replay unchanged.
- Version-4 worker artifacts are validated and applied to a private staging worktree. Strict Review JSON can produce approval, valid findings, or an unverified outcome. Only valid findings can start a fresh bounded Repair sibling.
- Foreground applies an approved cumulative candidate through a two-phase checkpoint transaction. Process-lifetime background stops at `ready_to_apply` and exposes scoped list/status/wait/cancel/resume/apply controls to the parent.
- A private sequenced team journal, cross-process parent/run owner leases, exact task/workspace/assignment binding, durable artifacts, startup orphan reconciliation, and conservative apply recovery make interruption truthful. A dead process does not pretend its compute continued.
- The role router is provider-neutral and records eligibility/fallback decisions. Production currently supplies only the immutable parent pin, so multiple live role-specific models are not claimed.
- Usage, cost coverage, review/repair rounds, lifecycle, evidence, and safe failures are normalized into durable state and parent-visible events. Unknown cost remains unknown. The private team journal and child session logs persist assigned prompts and tool calls; `/agents` projections and `agent_team_activity` omit prompts, patch bodies, credentials, account data, and private paths, while the general JSONL transcript can include prompts and tool arguments.
- The assembled E2E path proves parent delegation, isolated implementation, two reviewers, one repair, re-review approval, parent synthesis/apply, five completed depth-one children, durable activity, and complete worktree cleanup.

## Intentionally absent

Recurs does not yet implement a worker daemon, computation that survives CLI exit, depth beyond one, child-created children, automatic task decomposition, retries of terminal children, dirty-parent snapshots, multiple live role-specific backends, dynamic role libraries, Linux/Windows OS sandboxing, exact enforcement when cost telemetry is absent, auto-commit, push, PR, deploy, schedules, or the company desktop UI. Process-lifetime background becomes interrupted after process loss; resume is explicit and creates fresh sibling attempts. Worktree isolation and owner locks are application boundaries; the standalone macOS CLI separately applies Seatbelt containment to shell and verification children.
